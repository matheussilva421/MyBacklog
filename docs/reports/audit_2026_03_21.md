# Relatório de Auditoria e Correções - 21 de Março de 2026

## Visão Geral
Uma auditoria completa foi realizada em toda a codebase do **MyBacklog** para identificar bugs dinâmicos, código morto e lógicas desconexas, especialmente em processos críticos como o *Onboarding*.

## Bugs Identificados e Resolvidos

### 1. DexieError: "Table settings not part of transaction"
**O Problema**: Ao iniciar o sistema pela primeira vez com um banco de dados vazio, a função `seedDefaultLibrary` semeava a biblioteca básica. No decorrer do pipeline de transações estruturadas, o sistema acionava `syncStructuredRelationsForRecord`, que de forma encadeada executava verificações de "soft delete" nas tabelas. A checagem de exclusão lógica necessita ler do `db.settings` o ID do usuário para associar no JSON, porém, essa tabela não havia sido informada ao abrir a transação de escrita original.

**A Solução**: 
- Modificamos a assinatura do Dexie Transactions em `src/services/backlogRepository.ts` e `src/core/structuredDataSync.ts` para explicitar a injeção da tabela `db.settings`. 

### 2. Deleções globais corrompidas e Omissão de `db.pendingMutations`
**O Problema**: Todo o sistema foi refatorado recentemente para usar um fluxo Cloud Sync offline-first. Isso significa que *cada put/add* no LocalDB aciona uma chamada `enqueueMutation` que adiciona métricas ao `db.pendingMutations`. No entanto, 9 áreas críticas da aplicação e actions declaravam o `db.transaction()` sem incluir a `db.pendingMutations`, abortando o commit em massa.

**A Solução**:
- Em vez de uma alteração lenta, criamos e executamos um script iterador (`injectPendingMutations.cjs`) que filtrou o AST e varreu a codebase via expressões regulares injetando o objeto `db.pendingMutations` no array de qualquer `db.transaction()` no repositório. O processo varreu 9 arquivos salvando incontáveis falhas de salvamento em tela.
- Atualizamos o Mock nativo dentro de `syncTelemetry.test.ts` para evitar erros de arguments varArgs nos testes.

### 3. Falha Silenciosa de Onboarding e React 18 Strict Mode
**O Problema**: Mesmo o clique no `Iniciar Backlog` não apresentando mais o erro crítico do Dexie, a tela *não prosseguia* ao Dashboard, entrando num ciclo de renderização perpétuo em falso. Pior: nenhuma falha era reportada ao console de debug do vite.

**Causa**: O app contávamos com uma estrutura de tracking baseada no hook central `useBacklogDataState`. Ele utiliza o padrão `isMountedRef` para prevenir _race conditions_ entre requisições assíncronas concorrentes. Com o **React 18 StrictMode**, componentes em modo dev sofrem um unmount seguido de um remount imediato. O destrutor do useEffect inicializava `isMountedRef.current = false`, mas o componente reinstanciado NÃO o devolvia ao estado original de `true`. Com isso, toda requisição de `refreshData()` abortava sua injeção ao estado silensiosamente. 

**A Solução**: 
- Alteramos o `useEffect` nativo de lifecycle dentro de `src/hooks/useBacklogDataState.ts`, atribuindo de imediato `isMountedRef.current = true` para garantir que as batidas de read do IndexedDB gerassem um setState válido para repasse às outras chaves reativas do painel.
